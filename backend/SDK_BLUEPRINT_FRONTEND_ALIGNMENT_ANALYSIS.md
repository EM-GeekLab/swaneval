# EvalScope SDK 蓝图 × 前端计划整合执行分析

## 1. 文档目的

本文件用于回答一个核心问题：

在以 `backend/EVALSCOPE_SDK_INTEGRATION_PLAN.md` 为后端蓝图、结合前端现有页面与 API 计划的前提下，接入 EvalScope SDK 后，项目在当前阶段**最多能做到什么**，以及前端计划应如何调整为可落地版本。

本分析是“可执行”视角：每个结论都对应开发任务、依赖、验收标准。

## 3. 统一结论（管理层可直接使用）

1. 当前系统已经具备“可用的 MVP 对接闭环”

- Auth / Model 基础 CRUD / Dataset / Criteria / Tasks / Results 主链路可联调。

2. 接入 EvalScope SDK 后可显著提升“执行内核能力”，但不会自动补齐“平台能力”

- SDK 擅长：评测执行、报告产物。
- SDK 不负责：业务模型字段持久化、任务日志 SSE、队列监控、权限模型、前端交互语义一致性。

3. 前端计划总体方向合理，但必须分层承诺

- 本迭代承诺：任务执行与结果分析主流程。
- 下迭代承诺：实时日志、精细子任务可视化。
- 架构升级后承诺：队列/Worker/GPU 监控与企业级恢复。

---

## 4. SDK 接入后的能力上限（用于约束前端需求）

## 4.1 本阶段可承诺（建议前端可直接排期）

1. 任务创建与执行

- 后端将任务参数映射为 `TaskConfig` 并调用 `run_task`。
- 前端可稳定看到任务状态流转：`pending/running/completed/failed`。

2. 结果分析

- 可从 SDK `work_dir` 读取报告并回写本地结果表。
- 前端可稳定使用：Leaderboard / Summary / Error Results。

3. 评测参数基础配置

- temperature/max_tokens/top_p/repeat_count 等基础参数可用。

## 4.2 中等改造后可承诺（前端可先做 UI，后端补接口）

1. repeat_count 多子任务精确映射

- 需要后端在任务创建时预建 subtasks，并把 SDK 执行进度映射到 run_index。

2. 任务日志可视化

- 先做“日志快照轮询”版本，再升级 SSE 实时流。

3. 报告统一视图

- 先支持“读取 SDK 报告原始 JSON + 平台统一展示”。

## 4.3 当前阶段不可承诺（不建议前端本迭代绑定）

1. 多 worker/GPU 调度看板
2. 跨进程/跨机器强一致断点恢复
3. 完整企业报表导出工作流（DOCX/HTML 模板化 + 权限控制 + 审计）

## 5. 与前端计划的对齐矩阵（可直接用于需求评审）

| 前端模块      | 计划能力                | 与 SDK 关系                           | 当前可行性   | 处理建议                                       |
| ------------- | ----------------------- | ------------------------------------- | ------------ | ---------------------------------------------- |
| Models        | 新增字段 + 连通性测试   | SDK 无直接支持                        | 中           | 本迭代后端补模型字段迁移 + `/models/{id}/test` |
| Datasets      | 上传/预览/删除          | SDK 依赖输入质量                      | 高           | 本迭代补删除一致性（文件+版本）                |
| Criteria      | CRUD + 测试             | SDK 可消费 metric，不管理业务配置     | 高           | 保持现状，后续扩展 llm_judge/script 真实现     |
| Tasks         | repeat 可视化、暂停恢复 | SDK 提供执行，不提供业务 subtask 语义 | 中           | 本迭代统一 subtask 语义；先保证状态正确        |
| Task Detail   | 实时日志                | SDK 不直接提供 SSE API                | 低（本迭代） | 改为两阶段：先轮询日志，再 SSE                 |
| Results       | 榜单/汇总/错误分析      | SDK 结果可映射                        | 高           | 本迭代做稳定映射与字段标准化                   |
| Reports       | 生成与导出              | SDK 仅提供原始产物                    | 中低         | 先做 JSON 报告读取，导出下阶段                 |
| Queue Monitor | worker/GPU              | SDK 不提供平台调度监控                | 低           | 架构升级后再做，避免本迭代承诺                 |

## 6. 整合后的分阶段开发路线（后端+前端）

说明：本节保留中长期路线作为后续版本参考；本次交付以第 6.1 节的“24 小时冲刺方案”为唯一执行口径。

## 6.1 24 小时冲刺方案（本次必须交付）

目标：在 1 天内交付“前后端可调用 SDK 且功能覆盖尽可能多”的可演示成品。

交付定义（DoD）：

1. 前端可完成完整主链路

- 注册/登录 -> 新建模型 -> 新建数据集 -> 新建准则 -> 创建任务 -> 查看结果。

2. 后端默认走 SDK 执行链路

- `TaskConfig + run_task` 成为任务执行主路径（保留故障开关）。

3. 前端 UI 对齐 SDK 可用能力

- 支持基础参数与关键配置透传（temperature/max_tokens/top_p/repeat_count）。
- 明确显示“当前版本不承诺”的能力（实时 SSE 日志、GPU 调度看板、复杂导出）。

4. 当天联调通过

- 至少 1 条端到端任务在前端可见完成状态，且结果页可见榜单/汇总/错误样本。

### 6.1.1 24 小时排程（并行）

0-2 小时：后端接口硬断点清理

1. 实现 `POST /models/{id}/test`。
2. 模型扩展字段落地：`description/model_name/max_tokens`（含迁移）。
3. 快速回归：模型 CRUD + test。

2-6 小时：SDK 主链路固化

1. 任务执行默认切到 SDK 适配路径。
2. 结果入库稳定化：保证 `results/leaderboard/summary/errors` 均有数据可读。
3. repeat_count 语义落地（优先策略：预建 subtasks + 显式状态推进）。

2-6 小时（前端并行）：UI 对齐

1. Models 页面补扩展字段输入与“连通性测试”按钮。
2. Tasks 页面与详情页对齐 repeat/subtask 新语义。
3. Results 页面增强容错（空值、运行中状态、无结果提示）。

6-10 小时：删除一致性与联调稳定性

1. Dataset 删除补齐文件与版本记录清理。
2. 前后端联调并修复接口字段不一致问题。
3. 增加最小日志可见能力（若来不及 SSE，先提供状态信息和错误文本展示）。

10-14 小时：集成测试与演示数据准备

1. 运行 E2E：前端触发任务并完成落库。
2. 准备 1-2 套演示数据（成功任务、失败任务）。
3. 固化演示脚本与已知限制清单。

14-18 小时：回归与发布包整理

1. 后端关键接口冒烟。
2. 前端构建检查与关键页面手测。
3. 输出“可交付说明”：已实现能力、未实现能力、下一步计划。

### 6.1.2 本次范围控制（必须遵守）

必须做：

1. SDK 主执行链路可用。
2. Models 扩展字段 + 模型连通性测试。
3. Tasks/Results 页面与 SDK 输出语义对齐。
4. 端到端联调可复现。

可选做（有余力再做）：

1. 日志轮询接口。
2. 报告 JSON 统一展示。

明确不做（避免拖垮 1 天交付）：

1. SSE 实时日志全能力。
2. Queue/Worker/GPU 监控体系。
3. 企业级报告导出（docx/html/csv 完整链路）。

### 6.1.3 当天验收门槛（Go/No-Go）

Go 条件（全部满足才算交付）：

1. 前端可创建并成功运行 SDK 任务。
2. 任务状态在前端可见完整流转。
3. 结果页可见 leaderboard/summary/errors 数据。
4. 模型测试按钮可返回成功或失败信息。
5. 至少一条 E2E 运行记录可复现（含任务 ID 与结果条数）。

No-Go 条件（任一触发则不得宣称交付）：

1. 任务虽完成但结果未入库。
2. 前端页面无法消费新增字段导致报错。
3. 模型 test 接口不可用。

## Phase A（P0，2 周内，先完成可交付闭环）

目标：前端主流程稳定可演示，SDK 进入主执行路径。

### A1 后端任务

1. SDK 执行路径固化

- 以 `TaskConfig + run_task` 为默认执行策略（保留开关）。
- 完成结果导入与失败处理规范。

2. 模型能力补齐

- 实现 `POST /models/{id}/test`。
- 增加模型扩展字段：`description/model_name/max_tokens`（含迁移）。

3. 数据集删除一致性

- 删除 Dataset 时同步删除 upload 文件与版本记录。

4. 子任务语义修正

- 创建任务时预建 `repeat_count` 条 subtasks。
- SDK 执行时更新对应 subtask 进度或落地“汇总模式”明确约定。

### A2 前端任务

1. Models 页补字段输入/展示与“连通性测试”按钮。
2. Tasks 页/详情页按新的 subtask 语义渲染。
3. Results 页按已稳定字段做展示容错（空值/延迟字段）。

### A3 Phase A 验收标准

1. 从前端创建任务到结果落库全链路成功率 >= 95%（测试环境）。
2. 模型测试接口可稳定返回 `{ok, message}`。
3. repeat_count=3 时，任务详情可见 3 个子任务（或明确展示“汇总模式”）。
4. 删除上传数据集后，DB 与磁盘均无残留。

## Phase B（P1，后续 1~2 周，提升体验与可观测性）

目标：让前端“可解释”和“可观测”。

1. 任务日志两阶段实现

- B1：日志快照轮询 API（先可用）
- B2：SSE 实时日志接口（再优化）

2. 报告统一层

- 把 SDK `work_dir` 报告转换为平台标准 JSON。
- 前端报告页先支持查看与下载 JSON。

3. 角色权限落地

- 后端关键写接口接 `require_role`。
- 前端 Sidebar 按 role 过滤菜单。

验收：

- 任务执行中前端能看到持续日志更新。
- viewer 角色无法访问写接口，前端入口可见性正确。

---

## Phase C（P2，架构增强阶段）

目标：对齐原计划 Module 5/6 的企业级能力。

1. Queue/Scheduler 体系

- `queue/status`、`queue/workers`、worker/gpu 分配策略。

2. 报告导出体系

- docx/html/csv 导出链路与权限控制。

3. 高级恢复能力

- 跨进程断点续跑、一致性恢复、失败重放。

---

## 7. 关键技术决策（现在就要定）

1. 子任务语义决策

- 选项 A：严格每 repeat 一个 subtask（推荐，前端语义最清晰）。
- 选项 B：保持汇总 subtask（成本低，但前端体验受限）。

2. 日志通道策略

- 先轮询后 SSE，避免一次性引入复杂事件系统。

3. 结果映射策略

- 固定“SDK 产物 -> 本地结果”映射表，避免散落在业务代码中的硬编码解析。

---

## 8. 风险与缓解

1. 风险：SDK 报告结构随版本变化

- 缓解：锁定 evalscope 次版本；建立解析适配层与回归测试样例。

2. 风险：任务执行耗时影响 API 进程

- 缓解：逐步迁移到独立 worker，API 仅做调度与查询。

3. 风险：前端过早实现高阶监控页面，后端无法供给

- 缓解：按本文件能力分层做 feature flag 与里程碑验收。

## 10. 最终判断：前端计划是否合理

结论：合理，但本次必须按“24 小时冲刺范围”收敛交付。

- 合理的部分：任务与结果主链路、基础管理页、可视化分析页。
- 需要降级的部分：实时日志、复杂报告导出、队列/worker/gpu 监控。
- 决策建议：当天以 6.1 节 Go 条件为发布门槛；其余能力进入后续里程碑。
